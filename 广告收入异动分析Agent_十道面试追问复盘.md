# 广告收入异动分析 Agent：十道面试追问复盘

整理时间：2026-06-22  
适用岗位：AI 大模型策略产品经理 / Agent 产品经理 / 广告策略产品经理

## 总体判断

你的项目材料具备真实业务场景、明确 AI Agent 落地点、可量化结果和较强业务价值，但在真实面试中会被重点追问四类问题：

1. 指标定义是否严谨：例如“准确率 99%”到底评估什么、样本量多大、真值如何得到。
2. Agent 边界是否清楚：哪些由 LLM 负责，哪些必须由规则、工具和确定性代码负责。
3. 线上治理是否完整：灰度、监控、回滚、人工反馈、Eval 回归是否能闭环。
4. 架构理解是否扎实：WorkBuddy、MCP、Skill、RAG、Evals、模型替换等概念不能只会泛泛描述。

下面按今天已经追问的 10 道题逐题复盘。

## 问题 1：准确率 75% -> 99% 到底怎么定义？

### 面试官追问

你材料里写：

> 异动分析准确率由 75% 提升至 99%，AI 归因与人工分析一致性约 80%。

这两个数字之间有明显张力。如果归因一致性只有 80%，为什么准确率能到 99%？

需要回答：

- 评估对象是什么：异动识别、一级归因、二级归因，还是报告整体正确？
- 标注真值来自哪里？
- 样本量是多少，覆盖哪些业务线和时间范围？
- 99% 是离线评测、线上结果，还是某个子模块结果？
- 99% 和 80% 的关系是什么？

### 你的回答主要问题

你的回答能把“准确率”落到曝光 / CPM 归因，且解释 99% 是历史 T+1 分析、80% 是实时异动分析，这一点有效。

但存在三个硬伤：

1. 样本量表述不严谨。你说 3 月到 6 月、每天 3-10 条，总样本“千条左右”，按时间推算大概是 360-1200 条，跨度过大，显得没有掌握评测集规模。
2. “准确率”定义仍不够精确。没有说明是 Top1 命中、Top3 命中、一级归因命中、二级归因命中，还是多因子贡献排序正确。
3. 真值来源有偏差风险。历史产运报告本身可能有口径差异或错误，需要说明如何清洗、统一标签和人工仲裁。

### 标准答案方向

更好的回答应把准确率拆成多层：

1. L1 归因准确率：收入异动主要由曝光还是 eCPM / CPM 驱动。
2. L2 归因准确率：如果是 CPM，进一步判断是用户行为、行业预算、广告主结构、模型预估、扣费机制、策略规则等哪类因素。
3. L3 证据充分率 / 报告可采纳率：结论是否被关键指标支撑，是否能被产运直接采用。

可参考表达：

> 这里的 99% 不是指所有复杂根因都完全命中，而是指历史 T+1 异动分析场景下，Agent 输出的一级或二级核心归因与人工复盘结论一致的比例。我们把历史产运报告结构化为标签体系，例如曝光链路、CPM-用户行为、CPM-行业预算、CPM-广告主结构、CPM-预估模型、CPM-策略规则等。评测时要求模型输出主因、证据指标和贡献解释，只有主因命中且关键指标能支撑结论才算通过。实时场景因为当天数据不完整，归因链路缺少最终结算和部分广告主预算信息，所以一致性约 80%。

还应补充：

> 由于人工历史报告只是近似真值，我们做了标签标准化和人工抽检。对于多因子共同作用的 case，不强行要求单一 Top1，而是看主因排序和贡献解释是否合理。

## 问题 2：CPM 下降时，Agent 怎么判断主因？

### 面试官追问

如果某天收入下降 10%，曝光基本不变，CPM 下降 10%。同时：

- CTR 下降 3%
- CVR 下降 8%
- 某头部行业预算下降 15%
- pCTR 下降 5%
- 实际扣费 CPC 下降 6%

Agent 怎么判断主因？

需要回答：

- CPM 怎么拆成可计算因子？
- 每个因子对 CPM 下降的贡献怎么量化？
- 多因素同时变化时，怎么避免强行因果归因？
- 报告里怎么表达主因和不确定性？

### 你的回答主要问题

你能想到 eCPM 与 pCTR、pCVR、bid 相关，也能提到“自身变化”和“结构变化”，这是亮点。

但有四个问题：

1. 公式过于简化。广告 eCPM 不一定都能写成 `pCTR * pCVR * bid`，不同计费模式、竞价 / 合约、oCPX / CPC / CPM / CPA 都有不同口径。
2. 混淆实际 CTR / CVR 和 pCTR / pCVR。实际 CTR 是结果指标，pCTR 是模型预估指标，二者不一定同步变化。
3. 把排除 CTR / CVR 后的剩余部分统一归为行业预算，过度简化。剩余因素还可能包括竞争强度、出价策略、扣费机制、召回填充、排序策略、保护规则等。
4. “80% 主因阈值”没有解释来源，是业务经验、历史调参还是管理口径，需要说清楚。

### 标准答案方向

先按广告产品和计费模式拆开，再讲统一框架：

> 我不会直接让模型自由判断主因，而是先把收入拆成可解释链路。最上层是收入 = 曝光量 * eCPM / 1000。曝光基本稳定时，问题落到 eCPM。对竞价广告，eCPM 可以进一步拆成流量结构、候选广告结构、预估质量、出价 / 预算、排序与扣费机制等模块；在 oCPX 场景下，排序侧近似受 pCTR、pCVR、bid 或目标成本影响，但不同计费模式要使用不同公式，不能一刀切。

贡献量化可以这样回答：

> 量化上我会做两类拆解。第一类是指标贡献拆解，例如用对数差分、顺序分解或 Shapley 方法，把 eCPM 下降分摊到 pCTR、pCVR、bid、扣费率、行业结构等因子上。第二类是结构拆解，例如行业 / OG / 广告主维度，用 mix effect 和 within effect 区分“单个行业 eCPM 下降”与“低 eCPM 行业占比上升”。最终不是只看相关性，而是要求贡献率、方向一致性、链路证据三者同时成立。

避免强行归因：

> 如果多个因子贡献接近，或者关键链路数据缺失，Agent 不输出唯一主因，而输出“多因子共同驱动”或“需人工确认”。报告里给出主因置信度、贡献排序、缺失数据和下一步排查建议。比如主因贡献超过 60% 或 Top2 累计超过 80% 才给明确结论，否则降级为候选原因列表。

## 问题 3：哪些判断由模型负责，哪些由规则 / 工具负责？

### 面试官追问

你提到通过 Prompt 约束、反思机制、校验规则解决幻觉、强行因果、口径漂移。

需要回答：

- 数据查询和指标计算，是否允许 LLM 自己写 SQL 或自由选维度？
- 贡献率计算，是模型算，还是工具算？
- 根因判断中，模型到底起什么作用？
- 校验规则拦截什么？举真实例子。
- 如果模型结论和规则计算冲突，怎么处理？

### 你的回答主要问题

你说不允许 LLM 自己写 SQL，而是人工写好 SQL 让模型执行，这是对的方向。你也提到数据校验和结论完整性校验。

最大问题是：

> 你说贡献率由模型计算。

这在真实面试里很危险。贡献率属于确定性数值计算，不应该交给 LLM 自己算。LLM 可以负责规划、解释和报告生成，但关键数值计算必须由工具完成。

另外：

1. “人工写 SQL，LLM 执行”应更准确地表述为“预置参数化 SQL / 指标工具，LLM 选择工具和填参数”。
2. 用 CDP 抓线上看板做数据校验可以作为过渡方案，但长期更稳定的是统一指标服务、数据 API 或元数据血缘校验。

### 标准答案方向

核心原则：

> LLM 不负责事实和数值的最终裁决。LLM 负责规划、工具选择、解释和报告生成；确定性数据查询、指标计算、贡献率拆解、阈值判断和一致性校验由工具 / 规则完成。

可参考回答：

> 我把链路分成工具层、策略规则层和模型推理层。数据查询不允许 LLM 自由写 SQL，而是封装成参数化工具，例如按业务线、广告位、日期、竞价 / 合约拉取收入、曝光、eCPM、CTR、CVR、行业、广告主等指标。LLM 只能选择工具和填参数，不能自由拼字段。指标计算和贡献率计算也不交给模型，而是由确定性工具完成。模型主要负责三件事：决定下一步下钻路径；把工具输出的贡献排序和业务知识结合形成候选根因；生成可读报告，并说明证据、不确定性和下一步排查建议。

校验规则：

> 校验规则主要拦截三类问题：数据不一致、计算不闭环、结论无证据。例如，Agent 拉取的收入总额必须和线上看板在同口径下误差小于阈值，否则中止分析；报告中如果说 CPM 是主因，则必须给出 CPM 变化幅度、对收入变化的贡献率，以及下钻到 CTR / CVR / 行业预算 / 广告主结构的证据。

冲突处理：

> 如果模型结论和规则冲突，默认规则优先，Agent 不输出确定结论，而是降级为“候选原因 + 冲突点 + 需人工确认”。只有人工复盘确认规则覆盖不到新 case 后，才把这个 case 纳入知识库和规则迭代。

## 问题 4：RAG 怎么避免历史经验把模型带偏？

### 面试官追问

历史知识库里有一条案例：

> 某业务 CPM 下降，历史上通常是教育行业预算下降导致。

但今天教育行业预算没下降，反而是游戏行业占比上升且 eCPM 较低。RAG 可能把模型带偏。

需要回答：

- 知识库存什么？
- 检索召回用什么 key？
- 怎么避免历史 case 覆盖当前数据事实？
- 检索出来的知识是参考信息还是强规则？
- 如何评估 RAG 是正向提升，而不是引入旧经验偏差？

### 你的回答主要问题

你能把知识库分成“基础概念”和“历史 case”，并说明基础概念是强规则、历史 case 是参考信息，这是对的。

但问题在于：

1. 召回机制只说语义相似，不够可靠。异动分析需要结构化过滤，例如业务线、广告位、异常指标、竞价 / 合约、行业、归因类型。
2. 避免带偏只靠 Prompt 约束“必须由数据推导”，这太软，需要证据优先级和适用条件校验。
3. RAG 效果没有量化，只说“结论更定制化”，不是面试级评估指标。

### 标准答案方向

知识库分三类：

> 第一是强规则类，包括指标口径、业务线流量范围、维度映射、SQL 参数约束；第二是归因规则类，包括曝光链路、CPM 拆解、竞价 / 合约差异、异常阈值；第三是历史 case 类，包括历史异动原因、对应证据、适用条件和排除条件。历史 case 只作为参考，不作为结论依据。

召回机制：

> 检索不是纯语义召回，而是结构化过滤 + 向量召回。先用业务线、广告位、指标类型、竞价 / 合约、异常方向做 metadata filter，再在候选集合里做语义相似召回。历史 case 必须带适用条件，例如“教育行业预算下降且教育行业收入占比超过某阈值时适用”，否则只能作为背景信息。

避免带偏：

> Prompt 里明确证据优先级：当前数据 > 规则计算 > 基础口径 > 历史 case。历史 case 只能生成候选假设，不能直接生成结论。报告输出时，凡是引用历史 case 的结论，都必须验证当前数据是否满足适用条件；不满足则标记为“不适用”。

评估 RAG：

> 做 A/B 评估：无 RAG、只有规则 RAG、规则 + 历史 case RAG 三组。指标看一级归因准确率、二级归因准确率、证据充分率、幻觉率、平均分析耗时、人工采纳率，并单独统计“历史经验误导率”，即模型引用了历史 case 但当前数据不支持的比例。

## 问题 5：实时预警和 1 分钟下钻怎么设计？

### 面试官追问

如果今天 14:05 收入突然下降，Agent 要在 1 分钟内给出初步根因，链路怎么跑？

需要回答：

- 预警触发条件是什么？
- 实时数据延迟和不完整性怎么处理？
- 1 分钟内能下钻到什么深度，哪些要异步补充？
- 为了速度，预计算哪些指标？
- 14:05 报警，14:10 数据回补后结论反转，产品上怎么设计？

### 你的回答主要问题

你答到了告警等级触发、数据延迟标注、先出初步结论、回补后重跑，这是对的。

不足在于：

1. 触发条件太粗，只说同比 / 环比明显变化，没有说明基线建模、节假日、小时周期和误报控制。
2. 1 分钟下钻到广告主 / 广告位需要预聚合，否则实时拉明细不稳定。
3. 结论反转的产品设计不完整，没有提版本化、置信度、通知更新、审计日志。

### 标准答案方向

预警触发：

> 实时告警不是简单同比 / 环比，而是基于同星期同小时、近 N 天移动均值、节假日修正后的预测基线。触发条件可以是收入、曝光、eCPM 偏离预测值超过阈值，并且连续多个时间窗口成立，再按影响金额和偏离幅度分级。只有 P0 / P1 或连续告警才自动触发 Agent 下钻。

实时数据处理：

> 实时场景要区分快指标和慢指标。收入、曝光、请求、填充、pCTR、pCVR、出价、行业消耗可以分钟级；转化、ROI、深度转化等慢指标只作为待确认因素。报告必须带数据新鲜度和完整度，例如“截至 14:05，收入数据延迟 2 分钟，转化数据未完全回传”。

1 分钟下钻：

> 1 分钟内只做初步根因，不追求最终结论。下钻深度到业务线、广告位、竞价 / 合约、Top 行业、Top 广告主、曝光 vs eCPM。需要预计算分钟级收入、曝光、eCPM、行业 TopK、广告主 TopK、CTR / CVR / pCTR / pCVR、预算消耗、库存和请求漏斗。慢指标和长周期转化归因异步补充。

结论反转：

> 产品上要做结论版本管理。14:05 输出 v1 初步诊断，标记置信度和缺失指标；14:10 数据回补后自动生成 v2，并明确“结论是否变化、变化原因、旧结论是否失效”。如果旧结论影响了人工操作，需要通知相关群或工作台，并保留审计日志。

## 问题 6：项目除了提效，有没有商业价值？

### 面试官追问

你说释放 14h/周人工成本，并把 2-3 天缩短到分钟级。但这只是提效，不一定创造业务价值。

需要回答：

- 除了节省人力，还有没有直接或间接收入收益？
- 收益如何量化？
- 如果没有直接收入收益，怎么证明值得做？
- 北极星指标和过程指标是什么？
- 如果只能向老板汇报一个指标，选哪个？

### 你的回答主要问题

你能把价值上升到“减少异常期间收入损失 / 广告主赔付损失”，并考虑 Agent 成功率折损，这是加分点。

但两个问题比较明显：

1. “员工成长空间”不是强商业论证。可以作为组织效率补充，但老板更关心收入风险、SLA、规模化复用、口径一致性、专家经验沉淀。
2. 北极星指标选“准确率”不一定最优。准确率是门槛，但只看准确率会诱导系统少报、慢报、只报简单 case。

### 标准答案方向

商业价值：

> 这个项目的价值不只是节省 14h/周，而是降低广告收入异常的 MTTA 和 MTTR。以前从发现异动到定位原因可能要 2-3 天，现在分钟级给出初步根因，核心收益是减少异常持续时间、降低广告主赔付风险、提升收入波动治理能力。

收益量化：

> 业务价值 = 异常影响金额 / 分钟 * Agent 缩短的定位时间 * 归因可采纳率。  
> 如果涉及广告主赔付，则进一步计算：历史单位时长赔付金额 * 缩短时长 * 归因成功率。  
> 同时可以量化异常发现率、平均定位时长、人工介入率、报告采纳率，以及因提前发现而避免的潜在损失。

如果短期没有直接收入：

> 它仍然是风控和经营分析基础设施。价值包括：异常排查标准化、降低口径不一致、沉淀专家经验、减少对少数人的依赖、多业务线复用、为后续自动预警 / 自动诊断 / 自动处置打基础。

指标体系：

> 北极星指标可以选“可采纳根因覆盖率”或“有效异动处理时长下降”。过程指标包括告警召回率、误报率、平均检测延迟、平均下钻耗时、一级 / 二级归因准确率、证据充分率、人工采纳率、结论反转率、人工介入率、单位 case 成本。

向老板只汇报一个指标：

> 我会选“P0/P1 异动平均定位时长下降”，因为它最容易和业务风险、收入损失、广告主赔付建立关系。准确率必须作为门槛指标，例如二级归因准确率超过 95% 后，再汇报定位时长下降。

## 问题 7：灰度上线和线上监控怎么保证不大规模推错结论？

### 面试官追问

Agent 要上线给多个业务线使用，如何设计灰度上线和线上监控？

需要回答：

- 上线前离线评测集怎么构建？
- 灰度阶段怎么分流？
- 上线后监控哪些指标？
- 出错时怎么回滚或降级？
- 如何设计人工反馈闭环？

### 你的回答主要问题

你能按曝光漏斗、CPM 驱动因素、不同业务线和流量场景构建评测集，这是对的。

但问题包括：

1. 评测集缺少难度和风险分层，例如多因子、数据缺失、指标冲突、历史 case 误导、低样本波动、节假日周期。
2. 灰度不能只按业务线，还需要只读模式、人工确认模式、自动推送模式。
3. 上线门槛没有具体标准。
4. 你对“错误结论怎么回滚或降级”一开始答不出来，这是线上 AI 产品的大风险点。

### 标准答案方向

离线评测集：

> 上线前构建分层评测集。第一层是常规单因子 case，包括曝光漏斗、CPM、CTR / CVR、pCTR / pCVR、行业预算、广告主结构。第二层是复杂 case，包括多因子共同变化、贡献率接近、数据延迟、口径缺失、历史 case 容易误导、合约和竞价混合。第三层是高风险 case，来自历史 P0/P1 异常和人工复盘报告。每个 case 都有标准标签、关键证据和可接受结论范围。

灰度分流：

> 灰度分三阶段：只读影子模式、人工确认模式、自动推送模式。第一阶段 Agent 跟人工并跑，不影响业务决策；第二阶段结论给产运确认后再同步；第三阶段只对高置信、规则闭环的 case 自动推送。分流上按业务线、异常类型和风险等级逐步扩大。

线上监控：

> 监控分五类：链路稳定性、数据质量、结论质量、风险指标、业务效果。  
> 链路稳定性包括任务成功率、工具调用成功率、API 成功率、平均耗时、超时率、重试率。  
> 数据质量包括数据新鲜度、数据完整率、指标服务一致率、核心字段缺失率。  
> 结论质量包括一级 / 二级归因准确率、证据充分率、报告完整率、人工采纳率、结论反转率。  
> 风险指标包括错误结论率、高风险 case 自动推送率、人工拦截率、低置信结论占比。  
> 业务效果包括 P0/P1 平均定位时长、异常发现到确认时长、人工介入时长、重复问题解决率。

回滚 / 降级：

> Agent 要设计多级降级，而不是只有开关。第一级是结论降级：数据缺失、工具失败、贡献率不闭环时，不输出确定主因，只输出候选原因和需人工确认项。第二级是能力降级：如果 RAG 或模型不稳定，只保留确定性规则报告，例如收入、曝光、CPM、TopK 变化，不做自然语言归因。第三级是流量降级：某业务线错误率超过阈值，自动退回影子模式或人工确认模式。第四级是版本回滚：Prompt、工具、规则、知识库都要版本化；发布后错误结论率或结论反转率异常升高，回滚到上一版。

人工反馈闭环：

> 反馈不能只是“好 / 不好”，要结构化到错误类型：数据错误、维度选错、计算错误、主因判断错、证据不足、报告不完整、知识引用错误。反馈后不应直接写入 RAG，而是进入人工审核和标签化流程。通过审核后，分别更新规则、工具、Prompt、知识库或评测集。不是所有错误都适合靠 RAG 解决，计算错要修工具，字段口径错要修指标服务，归因边界错才更新规则或知识库。

## 问题 8：如何设计 Agent Eval 体系？

### 面试官追问

如果要把广告异动分析 Agent 做成公司内通用能力，不只服务 CSIG，怎么设计 Agent Eval？

需要回答：

- Eval case 怎么采样？人工历史报告够不够？
- 评分维度有哪些？
- 报告生成质量怎么评分？
- LLM-as-judge 能不能用？怎么避免 judge 胡说？
- Eval 怎么接入日常迭代，防止 Prompt / RAG / 工具升级后效果回退？

### 你的回答主要问题

你把稳定性、数据质量、结论质量、业务效果这些维度列出来了，但有几个关键问题：

1. 把线上过程指标当成 Eval case。工具调用率、接口成功率是监控指标，不是 eval case 本身。
2. 漏答“报告生成质量怎么评分”。
3. LLM-as-judge 的定位不准确。Judge 适合评估报告结构、证据引用、遗漏、矛盾，不适合评估接口成功率这类确定性指标。
4. “隔离上下文”不能防止 judge 胡说。Judge 需要受控输入，而不是信息越少越客观。

### 标准答案方向

Eval case 定义：

> Eval case 不等于线上日志指标。一个 eval case 应该包括：用户问题、业务线和时间范围、冻结的数据快照、工具调用期望、标准归因标签、关键证据、允许的等价答案、评分 rubric。人工历史报告可以作为初始来源，但不够，因为它偏向已发生且被记录的 case，还要补充人工构造的边界 case、数据缺失 case、多因子 case、历史经验误导 case，以及高风险 P0/P1 case。

评分维度：

> 分四层评分：任务完成、数据正确、归因正确、报告可用。任务完成看是否成功跑完、是否调用正确工具、耗时是否达标；数据正确看口径、时间范围、业务线、维度选择、计算结果是否正确；归因正确看一级归因、二级归因、贡献排序、证据闭环、不确定性表达；报告可用看结构完整、结论先行、数据支撑、风险提示、下一步建议、是否可被产运直接采用。

报告质量：

> 报告生成质量不是看文笔，而是看业务可用性：是否回答了收入变动幅度、主因、贡献率、影响范围、证据指标、置信度、待确认项和建议动作。可以人工评分 1-5 分，也可以让 LLM judge 做初筛，但关键 case 仍然人工抽检。

LLM-as-judge：

> LLM-as-judge 可以用，但只评估半结构化内容，比如报告是否引用证据、结论是否和给定数据矛盾、是否遗漏必填项。为了避免 judge 胡说，要给它固定 rubric、标准答案、数据摘要和 Agent 报告，并要求输出结构化评分和引用依据。关键准确率指标仍以确定性规则和人工标注为准，同时定期评估 judge 与人工的一致性。

接入日常迭代：

> 每次 Prompt、RAG、工具、指标口径升级，都跑固定回归集 + 最新线上失败集。设置 blocking gate，例如二级归因准确率不能下降超过 1pct，高风险 case 不能回退，数据一致率必须 100%。线上 badcase 经过标注后进入 eval 集，形成持续回归。

## 问题 9：WorkBuddy、MCP、Skill 的架构边界是什么？

### 面试官追问

你项目中用了 WorkBuddy + MCP + Skill。

需要回答：

- WorkBuddy、MCP、Skill 分别承担什么角色？
- 为什么需要 MCP？不用 MCP 直接调 API 不行吗？
- Skill 怎么定义和复用？
- 多业务线复用时，哪些通用，哪些定制？
- 如果换模型，哪些不变，哪些要重测？

### 你的回答主要问题

这题暴露比较明显：

1. WorkBuddy / MCP / Skill 边界讲混了。
2. 说 MCP 是“封装的一整套与大模型通信的 API 服务”，不准确。MCP 是把外部工具、数据源、资源以标准协议暴露给模型 / Agent 使用的连接层。
3. 说 MCP 比直接调 API 更快，这个理由不稳。MCP 不天然更快，价值在标准化、可复用、权限治理、工具发现、上下文协议统一。
4. Skill 复用没答出来。
5. 换模型后哪些要重测没答出来。

### 标准答案

架构角色：

> WorkBuddy 是用户入口和 Agent 运行平台，负责承载用户交互、任务触发、权限、执行记录和结果展示。MCP 是工具连接层，把数据库查询、指标计算、看板校验、知识检索等能力以标准工具形式暴露给 Agent。Skill 是任务级能力封装，定义“什么时候调用、按什么步骤执行、能用哪些工具、输出什么格式、如何校验结果”。

为什么需要 MCP：

> 不用 MCP，Agent 当然也可以直接调后端 API，但会有工具接入方式不统一、权限和审计难管、工具描述和参数 schema 分散、不同 Agent 或不同模型复用成本高等问题。MCP 的价值是把工具能力标准化，让模型通过统一协议发现和调用工具，同时便于权限、日志、版本和复用治理。它不是为了让调用更快，而是为了让工具生态更可控、更可扩展。

Skill 定义：

> 一个 Skill 至少包括：适用场景 description、触发条件、输入参数、分析流程、可调用工具列表、业务规则、输出模板、异常处理、校验规则和示例 case。比如 `csig_revenue_analysis` 会定义：先确认业务线和时间范围，再拉收入 / 曝光 / eCPM，再做竞价 / 合约拆分，再做曝光或 CPM 下钻，最后输出主因、贡献率、证据和不确定性。

多业务线复用：

> 通用部分是收入异动分析框架、指标计算工具、贡献率拆解、报告模板、校验规则和 eval 体系。定制部分是业务线流量范围、广告位映射、特殊策略规则、行业分类、数据源字段、历史 case 和阈值。产品化上应该把通用流程做成 base skill，把业务差异做成配置和知识库，而不是复制多份 skill。

换模型时：

> 架构上工具层、指标计算、数据 API、规则校验、权限审计、评测集不需要变。需要重测的是模型的工具调用稳定性、指令遵循、长上下文能力、RAG 使用方式、报告生成质量、对不确定性的表达、以及复杂多步任务完成率。推理模型可能更适合复杂归因，但成本和延迟更高，所以要重新评估准确率、耗时和成本 trade-off。

## 问题 11：相关性和因果归因怎么区分？

### 面试官追问

你前面多次提到“强行建立因果关系”。在广告收入异动分析里，很多指标只是相关，不一定因果。

需要回答：

- 在 Agent 里，哪些结论可以称为“因果”，哪些只能称为“相关 / 贡献解释”？
- 如果 CTR 下降和 CPM 下降同时发生，怎么判断 CTR 是原因，而不是共同受另一个因素影响？
- 有没有可能做真正的因果验证，例如实验、反事实、DID、因果图、策略日志？
- 报告里如何措辞，避免把相关性包装成因果？

### 你的回答主要问题

你能意识到：单链路、确定性公式推导更接近因果，多因素共同作用时更适合叫贡献解释。这是对的。

但第 2 点问题较大。你认为“点击是曝光后续环节，不会存在同一个因素共同作用到 CTR 和 CPM”，这个判断不成立。

广告系统里非常可能存在共同原因同时影响 CTR 和 CPM，例如：

- 流量结构变化：低质量流量占比上升，会导致 CTR 下降，同时竞价竞争和 eCPM 下降。
- 广告主结构变化：某类广告主退出或预算下降，剩余广告主素材质量、出价、转化预估都变化。
- 排序策略 / 召回策略变化：改变候选广告集合，既影响用户点击，也影响胜出广告的 eCPM。
- 模型版本变化：pCTR / pCVR 校准变化，会影响排序、胜出广告、扣费，也可能改变实际 CTR。
- 广告位样式或前端形态变化：会影响点击行为，也可能影响广告主竞争和出价效率。

另外，“CPM 在曝光环节已经完成，所以不会受点击影响”这个说法也要小心。对 CPC / oCPX 广告来说，单次曝光的预期 eCPM、排序、最终收入、扣费和转化回传之间不是简单的前后隔离关系。点击发生在曝光后，但排序和预估使用的是点击 / 转化概率；实际 CTR 变化也可能反映排序质量或流量质量变化。

### 标准答案方向

更稳的回答：

> 在这个 Agent 里，我不会轻易把结论称为因果。只有在两类情况下会接近因果判断：第一，指标之间存在明确的业务公式或系统链路，例如收入 = 曝光 * eCPM / 1000，曝光下降在口径闭环下可以解释收入下降的贡献；第二，有明确外部事件或策略日志，例如某广告位下线、某策略发布、某模型版本切换，并且时间、影响范围、方向和指标变化一致。  
> 对于 CPM、CTR、CVR、预算、出价、预估模型这类互相耦合的指标，我更倾向于表述为“贡献解释”或“候选原因”，而不是直接说因果。

关于 CTR 和 CPM 同时下降：

> CTR 下降和 CPM 下降同时发生时，不能直接判断 CTR 是 CPM 下降的原因。需要排查是否存在共同原因，例如流量结构变化、广告主结构变化、召回候选变化、模型版本变化、广告位形态变化等。判断时要看链路证据：CTR 下降是否集中在某业务线 / 广告位 / 行业 / 广告主；pCTR 是否同步变化；bid、预算、竞争广告数、扣费率是否变化；是否有策略发布或流量切换事件。如果只有相关变化，没有链路证据，就不能称为因果。

真正的因果验证：

> 可以做几类验证。第一是实验，例如对策略变更做 A/B，观察实验组和对照组 CPM、CTR、收入变化。第二是反事实，对同业务线、同广告位、同时间段，用历史基线或相似流量构造 counterfactual。第三是 DID，在策略只影响部分流量时，用未受影响流量作为对照。第四是结合策略日志、模型版本、广告主预算变更日志做事件归因。  
> 但多数日常异动分析无法完成严格因果验证，因此 Agent 报告应输出“最可能原因 + 证据 + 置信度 + 待验证项”。

报告措辞：

> 不说“CTR 下降导致 CPM 下降”，而说“当前数据表明，CPM 下降与 CTR / pCTR 下滑、低 eCPM 行业占比上升同时发生。其中 CTR 相关贡献约 xx%，但由于存在流量结构变化，暂不能判断 CTR 是唯一因果原因。建议进一步核查对应广告位的流量结构和模型版本变更。”

一句话总结：**能用公式和外部事件闭环的，才接近因果；大多数多因子广告归因，只能先叫贡献解释和候选原因。**

## 问题 12：反思机制到底怎么实现？

### 面试官追问

你前面说 Agent 能做“反思机制”，检查结论是否被关键指标支撑。

需要回答：

- 是同一个模型再读一遍自己的报告，还是用独立 judge？
- 反思时检查哪些项目？列出检查清单。
- 反思发现问题后，是自动重跑、补充取数、降级输出，还是拦截给人工？
- 反思机制带来了哪些可量化收益？

### 你的回答主要问题

你这题答得基本及格。你能说清楚：

- 用独立 judge，而不是同一个模型简单自查。
- 检查核心指标完整性、报告完整性、逻辑链条。
- 根据问题类型触发重取数、重写报告、补充数据或降级输出。
- 引入反思后，报告完整率和最终任务成功率提升。

需要修正三点：

1. 你有一个口误：“加入反思机制后，能够明显降低输出报告的完整度”，应立即改成“提升报告完整度”。
2. Judge 的输入和权限没讲清楚。独立 judge 不能只读报告，它至少要看到用户问题、工具输出的数据摘要、报告、必填项 checklist 和评分规则。
3. “反思机制让取数成功率接近 100%”表达不严谨。不是 judge 自己提升了取数能力，而是 judge 检测到缺数后触发重试 / 补取数，使最终任务成功率提升。

### 标准答案方向

可参考回答：

> 我们的反思机制不是让同一个模型简单重读一遍报告，而是独立 judge 链路。Judge 的输入包括用户问题、工具调用日志、关键指标数据摘要、Agent 初版报告、必填检查清单和评分 rubric。它不重新做业务归因，而是判断报告是否满足输出标准，以及结论是否被当前证据支撑。

检查清单：

- 数据完整性：收入、曝光、eCPM、竞价 / 合约、核心下钻维度是否齐全；数据时间范围和业务线是否正确。
- 计算闭环：收入变化是否能被曝光和 eCPM 贡献解释；贡献率是否加总闭环；单位和同比 / 环比口径是否一致。
- 结论证据：报告中的主因是否有对应指标支撑；是否存在方向相反的数据；是否把历史 case 当成当前事实。
- 报告完整性：是否包含异动幅度、主因、贡献率、影响范围、风险提示、待确认项和下一步动作。
- 不确定性表达：数据缺失、多因子贡献接近、慢指标未回传时，是否明确降级为候选原因。

发现问题后的处理策略：

> 如果是数据缺失，自动补取数或重试工具；如果是计算不闭环，调用计算工具重算；如果是报告格式缺失，基于已有数据重写；如果是结论与数据冲突，则拦截确定性结论，降级为候选原因并提示人工确认；如果缺少关键慢指标，则输出临时版本并标记等待回补。

收益表达：

> 引入反思后，报告完整率从约 50% 提升到接近 100%；数据缺失导致的失败 case 明显下降，最终任务成功率接近 100%；同时结论无证据、遗漏关键指标这类 badcase 显著减少。这里提升的不是 judge 自己的归因能力，而是通过检查和重试机制提高了最终可交付质量。

## 问题 13：扩展业务线后如何控制成本和延迟？

### 面试官追问

如果 Agent 扩展到更多业务线，调用更多工具、RAG 和 judge，成本和延迟都会上升。

需要回答：

- 哪些步骤必须用大模型，哪些可以用规则 / 小模型 / 缓存？
- 如何控制 token 成本和工具调用成本？
- 什么时候用通用大模型，什么时候用推理模型？
- 如果老板要求“1 分钟内出报告，成本下降 50%”，怎么取舍？

### 你的回答主要问题

你答到了缓存、失败阈值、不同环节用不同模型，这是对的。

但有几个问题：

1. “执行工具流程、输出结论由小模型承接”不稳。工具执行本身不需要模型，应该由程序执行。小模型可以做简单路由、格式化、摘要，但不能承担关键归因结论。
2. 通用大模型和推理模型的适用场景说反了一部分。意图理解和简单工具选择一般不需要推理模型，用通用模型甚至规则分类就够了；复杂多因子归因、冲突证据判断、低置信 case 复核，才适合推理模型。
3. “工具调用 3 次失败后用已取数据输出最终结论”有风险。数据缺失时不能输出最终结论，只能输出临时诊断 / 候选原因。
4. 用更多 RAG 文档减少 token 的逻辑不一定成立。RAG 也会带来检索和上下文成本。降 token 的关键是结构化摘要、metadata filter、只召回必要片段、缓存常用口径。

### 标准答案方向

链路拆分：

> 我会把 Agent 链路拆成确定性层、轻模型层和强模型层。确定性层包括取数、指标计算、贡献率拆解、阈值判断、数据一致性校验，这些不用大模型。轻模型或规则层用于意图分类、业务线识别、工具路由、报告模板填充。强模型用于复杂归因解释、证据冲突判断、不确定性表达和最终报告润色。推理模型只用于高风险、复杂、多因子或规则冲突的 case，不作为默认链路。

成本控制：

> 控制成本主要靠四件事。第一，预计算和缓存，把分钟级收入、曝光、eCPM、TopK 行业 / 广告主、常用口径提前算好。第二，工具调用规划，先粗粒度定位，再按 TopK 下钻，避免全量维度扫描。第三，RAG 做 metadata filter 和短摘要，只召回业务线、指标、异常类型匹配的少量知识。第四，Prompt 和报告模板结构化，减少长上下文和重复解释。工具失败时不无限重试，超过阈值后降级为临时诊断，而不是输出最终结论。

通用模型 vs 推理模型：

> 通用大模型适合常规意图理解、普通报告生成、低风险归因解释；推理模型适合复杂 case，例如多因子贡献接近、数据冲突、历史 case 与当前数据不一致、P0/P1 高风险异常、模型结论和规则冲突。是否调用推理模型由风险等级和复杂度触发，而不是全链路默认使用。

1 分钟 + 成本下降 50% 的取舍：

> 如果老板要求 1 分钟内出报告且成本降 50%，我会做分层输出：30 秒内给 v1 快速诊断，只覆盖收入、曝光、eCPM、竞价 / 合约、TopK 行业 / 广告主；1 分钟内给可采纳初步结论；慢指标和复杂归因异步补充 v2。成本上优先做预计算、缓存、TopK 下钻、规则优先、模型路由分级，减少 RAG 召回长度和 judge 触发比例。取舍上，低风险 case 不走推理模型和完整 judge，高风险 case 保留完整校验，保证关键场景可靠性。

核心原则：**降成本不是简单换小模型，而是把确定性工作从模型里拿出去，把强模型留给真正需要判断的地方。**

## 问题 14：如何从广告异动分析复用到会员收入 / 电商 GMV？

### 面试官追问

如果公司另一个团队也想用你的 Agent 做“会员收入异动分析”或“电商 GMV 异动分析”，你怎么判断能不能复用？

- 业务规则和阈值。比如电商大促、会员促销、广告预算 pacing 的异常判断完全不同。
- 知识库和历史 case。
- Eval 集和人工标注标准。

平台设计：

> 我会做成“异动分析 Agent 平台”，而不是复制多个 Skill。平台包含通用引擎、指标树配置、工具适配器、知识库管理、报告模板、评测系统和监控系统。每个业务线接入时提交一个业务配置包，包括指标树、数据源映射、异常阈值、归因规则、可下钻维度、知识库和评测集。Agent 根据业务配置动态选择工具和分析路径。

接入新业务线前要求对方提供：

- 核心指标定义和指标树。
- 业务公式和可解释链路。
- 数据表 / API / 看板口径。
- 可下钻维度：业务线、渠道、场景、用户群、商品、广告主等。
- 历史异动报告和复盘 case。
- 策略变更 / 活动 / 版本发布日志。
- 异常等级和告警阈值。
- 人工验收人和标注标准。
- 数据权限和安全要求。

一句话总结：**复用的不是广告归因逻辑，而是异动分析的方法论和 Agent 工程框架。**

## 问题 15：RAG 和工具调用结果冲突时怎么办？

### 面试官追问

假设工具算出：

> 游戏行业贡献了 CPM 下降的 65%。

但 RAG 检索到历史 case：

> 该广告位过去 80% 的 CPM 异动都来自教育行业预算波动。

模型最终却写成“教育行业预算下降是主因”。

需要回答：

- 这是哪类 badcase？
- 根因可能出在哪里：召回、排序、Prompt、工具结果表达、模型遵循、校验？
- 怎么修？
- 如何设计规则，保证当前数据事实优先于历史经验？

### 你的回答主要问题

你识别到了“历史 case 覆盖当前数据事实”，这是对的。

但需要修正：

1. 不一定叫“知识库污染”。历史 case 本身可能是正确的，只是不适用于当前 case。更准确叫 RAG 误导、历史经验过拟合、证据优先级反转。
2. 根因分析太单一。你主要归因到模型遵循，但它可能同时出在召回、排序、Prompt、工具结果表达和校验。
3. 修复方案太软。只把 RAG 放到当前数据判断之后不够，还要做召回过滤、适用条件校验、证据优先级、输出校验。
4. “把规则写到 Agent 主文件”不够产品化。规则应该进入系统约束、校验器和 Eval。

### 标准答案方向

Badcase 定义：

> 这是典型的 RAG 误导型 badcase，本质是历史经验覆盖了当前数据事实，也可以叫证据优先级反转。历史 case 不一定是脏数据，它可能在过去成立，但当前数据不满足适用条件，所以不能作为结论依据。

根因拆解：

> 我会从六个环节排查。第一，召回是否缺少 metadata filter，比如业务线、广告位、异常指标、行业、时间范围、归因类型。第二，排序是否把历史 case 放得过前，导致模型注意力偏移。第三，Prompt 是否明确规定当前数据 > 工具计算 > 规则口径 > 历史 case。第四，工具输出是否结构化突出，例如“游戏行业贡献率 65%，Top1”。第五，模型是否违反指令，把参考 case 当事实。第六，校验器是否拦截了“报告主因与工具 Top1 贡献不一致”的情况。

修复方案：

> 我会做四层修复。召回层：历史 case 必须带 metadata 和适用条件，先结构化过滤，再语义召回。Prompt 层：明确历史 case 只能用于生成候选假设，不能覆盖当前数据事实。工具表达层：把当前数据的 Top 贡献因子结构化输出，并标记为 hard evidence。校验层：如果最终报告主因不是工具计算 Top1 或 Top2，必须要求模型解释差异；解释不成立则拦截或降级。

规则设计：

> 规则不是只写在主 Prompt，而是固化到校验器和 Eval 中。例如：当工具计算显示某因子贡献率超过 50%，且数据完整性通过时，报告主因必须与该因子一致；如果不一致，必须输出“冲突说明 + 人工确认”，不能输出确定结论。历史 case 引用必须满足适用条件，否则只能放在背景参考，不得进入主因。

一句话总结：**RAG 是假设来源，不是事实来源；当前数据和确定性工具结果才是事实来源。**

### 你的回答主要问题

这题回答不及格，主要问题：

1. “知识地图”只解决解释，不解决约束。指标口径不能只放给模型读，必须进入元数据管理、工具 schema、权限、校验规则和报告模板。
2. “固定字段名”不够。同一个字段名也可能在不同数据源、时间粒度、结算状态下含义不同。需要 metric_id、口径版本、数据源、粒度、更新时间、适用场景。
3. 报告里“直接展示口径”太粗。报告需要展示关键口径、数据源、时间范围、数据新鲜度、是否预估 / 结算、是否原始 / 有效。
4. “引入第三方大模型”判断谁可信是错误方向。两个数据源冲突不能交给另一个模型裁决，而要靠权威源优先级、数据血缘、口径定义、数据新鲜度、业务场景和人工仲裁。

### 标准答案方向

指标口径管理：

> 我会建立统一指标字典 / 指标元数据管理，而不是只做知识文档。每个指标都有唯一 metric_id，包含指标名称、计算公式、数据源、字段映射、时间粒度、更新频率、是否预估 / 结算、是否原始 / 有效、适用场景、口径版本、负责人和血缘关系。  
> 比如 `revenue_estimated_realtime` 和 `revenue_settled_finance` 是两个不同 metric_id，不能都叫“收入”；`raw_exposure` 和 `valid_exposure` 也必须拆开。

工具调用约束：

> 工具层不让模型自由选字段，而是按分析场景提供参数化指标工具。比如实时预警默认使用实时预估收入和原始 / 准实时曝光；T+1 复盘使用离线有效曝光和标准经营收入；财务对账使用结算收入。工具 schema 里要明确 metric_id、source、granularity、freshness、口径版本。模型只选择业务场景和时间范围，具体字段由工具映射，避免模型混用口径。

报告口径暴露：

> 报告正文不堆所有口径，但关键结论必须附口径说明。例如：“本次分析使用 Adata T+1 预估收入、离线有效曝光，数据截至 2026-06-22 10:00，非财务结算口径。”如果是实时分析，要明确“当前为预估口径，可能随回补变化”。如果跨口径比较，必须标注“不可直接对比”或转成统一口径后再分析。

数据源冲突处理：

> 先根据场景确定权威源。实时告警看实时看板，因为它时效最高；T+1 经营复盘看离线指标服务；财务结算看财务看板。如果同一场景下两个源不一致，先看数据新鲜度、口径版本、时间粒度、过滤条件和血缘链路；超过误差阈值则不输出确定结论，降级为数据质量异常，并提示需要数据 owner 确认。Agent 不应该自己判断谁对，而是根据预定义优先级和对账规则处理。

一句话总结：**没有指标口径治理，所有归因准确率都站不住。**

## 问题 17：Agent 结论和产运经验冲突时怎么办？

### 面试官追问

Agent 报告显示：某广告位 CPM 下降主要来自游戏行业预算下降，贡献率 70%。

但产运同学说：

> 我看广告主群里没人反馈预算下降，销售也说游戏行业没问题，你这个结论肯定错了。

需要回答：

- 先相信 Agent 还是产运？为什么？
- 如何组织排查，验证到底谁对？
- 如果 Agent 是对的，怎么推动产运接受？
- 如果 Agent 错了，怎么修复产品和评估体系？

### 你的回答主要问题

你这题答得及格偏上。你没有盲信 Agent，也没有盲信产运，而是强调对比双方分析流程、定位差异点，并区分口径差异和流程差异，这是对的。

还可以更强的点：

1. 排查路径不够具体。应该说清楚先查哪些证据：数据源、口径、贡献率计算、行业预算定义、广告主 TopK、销售口径、预算消耗日志、投放状态、出价、pacing 等。
2. 推动产运接受的表达要更谨慎。不要说“优先采用 Agent 分析结果”，而要说建立共同验收标准和证据链。
3. 修复评估体系不只是把 case 加入 eval，还要标注错误类型，并判断该修工具、规则、知识库、Prompt、指标口径，还是评测标签。

### 标准答案方向

基本立场：

> 我不会先相信某一方，而是把它当成“Agent 结论与人工经验冲突”的高价值 case。Agent 有数据链路优势，但可能缺业务上下文；产运有经验优势，但可能受历史经验和外部反馈影响。我的目标不是证明谁对，而是建立可复核证据链。

排查路径：

> 第一，确认口径：Agent 使用的是哪个收入、曝光、CPM、行业预算口径，是否与产运 / 销售一致。  
> 第二，复核工具结果：游戏行业贡献率 70% 是怎么分解出来的，Top 广告主、广告位、时间窗口是否正确。  
> 第三，看业务事实：游戏行业预算、消耗、出价、pacing、投放状态、广告主 TopK 是否变化。  
> 第四，看外部信息：销售说“没问题”是指整体预算没降，还是头部广告主没反馈？有没有小广告主、计划级、账户级预算变化被忽略？  
> 第五，看替代解释：是否是流量结构、模型预估、扣费机制、合约挤压导致被误归为游戏行业预算下降。

如果 Agent 对：

> 我会把 Agent 的证据链透明化，而不是直接说人工错了。比如展示游戏行业 Top 广告主预算 / 消耗变化、贡献率分解、影响广告位和时间窗口，并说明为什么销售反馈没有覆盖到这个变化。然后把这个 case 沉淀为新的分析规则或历史 case，让产运以后遇到类似情况能复用。

如果 Agent 错：

> 先做错误归因：是数据源错、字段口径错、贡献率公式错、行业映射错、工具调用错、RAG 误导，还是报告表达过度确定。不同错误对应不同修复：数据错修指标服务，公式错修计算工具，行业映射错修维表，RAG 误导修召回和校验，表达过度确定修 Prompt 和输出模板。最后把 case 加入回归 Eval，确保之后版本不再犯同类错误。

一句话总结：**冲突不是靠职位或经验裁决，而是靠证据链、口径和可复现计算裁决。**

## 问题 18：从分析 Agent 到处置 Agent，产品边界怎么设计？

### 面试官追问

下一阶段老板希望 Agent 不仅能分析，还能自动给出处置建议，甚至执行处置，例如：

- 某广告位曝光下降，建议回滚某策略；
- 某行业预算下降，建议通知销售跟进；
- 某模型预估异常，建议切回上一版模型；
- 某流量质量下降，建议调整召回或降级策略。

需要回答：

- 哪些处置可以自动执行，哪些必须人工确认？
- 自动处置前需要满足哪些条件？
- 如何设计审批、权限和审计？
- 如果自动处置导致收入进一步下降，责任和回滚机制怎么设计？

### 你的回答主要问题

你抓住了几个关键点：高风险策略 / 模型回滚必须人工确认，自动处置前要有可追溯证据链，处置后要监控和回滚。这些方向是对的。

但需要修正：

1. “召回和降级策略可以自动执行”风险较高。召回、降级策略也可能影响收入、广告主体验和用户体验，不一定是小范围。能否自动执行不是看动作名称，而是看影响半径、可逆性、置信度、历史验证和权限等级。
2. 自动处置前条件不够完整。除了可追溯，还要有置信度阈值、数据完整性、规则命中、影响范围、回滚方案、灰度范围、人工 on-call 可达等条件。
3. 责任归属表达要避免“Agent 设计者背锅”。企业里更合理的是系统责任边界：执行权限、审批人、发布人、owner、审计日志、事故复盘。

### 标准答案方向

自动 vs 人工：

> 我会按影响半径和可逆性分级，而不是按动作名称简单判断。低风险、可逆、影响范围小的动作可以自动执行，例如发送通知、创建工单、补充诊断、触发重跑、切换到只读分析、对单个低风险实验流量做预设降级。中风险动作需要人工确认，例如调整某个小流量广告位配置、暂停低优先级实验、通知销售跟进重点广告主。高风险动作必须人工审批，例如模型版本回滚、核心排序策略切换、大流量召回策略调整、影响收入分配或广告主消耗的动作。

自动处置前条件：

> 自动处置必须同时满足几个条件：数据完整性通过；根因置信度达到阈值；命中预先定义的处置 playbook；影响范围在白名单内；动作可逆且有回滚方案；有预估收益和风险评估；有实时监控指标；处置后能在短窗口内判断是否有效。否则只能给建议，不能执行。

审批、权限、审计：

> 设计 RBAC 权限和分级审批。Agent 只拥有低风险动作执行权；中高风险动作由业务 owner、策略 owner、数据 owner 或 on-call 审批。所有动作必须记录：触发原因、数据证据、模型版本、Prompt / Skill 版本、工具输出、审批人、执行时间、影响范围和回滚条件。审计日志用于事故复盘和权限追责。

收入进一步下降时：

> 每个自动处置都要有 guardrail 和自动回滚条件，例如处置后 5-10 分钟内收入、曝光、eCPM、投诉、广告主消耗等指标继续恶化超过阈值，则自动回滚到上一配置，并通知 owner。事故复盘时不把责任简单归给模型，而是看系统是否越权、审批是否缺失、playbook 是否错误、监控是否失效、回滚是否及时。修复时把 case 加入禁用规则、评测集和处置 playbook 更新。

一句话总结：**分析 Agent 可以大胆给建议，处置 Agent 必须保守执行、分级授权、可逆、可审计。**

## 问题 19：升级 Agent 时，选更强模型、微调，还是工具治理？

### 面试官追问

公司可以选三种方案升级 Agent：

A. 换更强的通用大模型。  
B. 用历史异动 case 做 SFT / 微调。  
C. 保持当前模型，但加强工具、规则、Eval 和数据治理。

资源只能选一个，你选哪个？为什么？

需要回答：

- 你的选择是什么？
- 另外两个为什么不优先？
- 用什么指标证明选择正确？
- 什么情况下会改变选择？

### 你的回答主要问题

这题你答得比较亮眼。你优先选 C 是合理的：对于强数据、强口径、强工具链的异动分析 Agent，瓶颈通常不是“模型不够聪明”，而是工具化、数据治理、规则校验、Eval 体系没有打牢。你也能解释 A 成本高收益不确定，B 依赖高质量结构化样本。

需要补强两点：

1. “全公司复用就必须换更强模型”不一定成立。全公司复用更需要平台化、配置化、数据治理和 Eval。只有当跨业务线复杂推理、长上下文、多工具规划成为主要瓶颈时，才优先换模型。
2. 证明选择正确的指标可以更具体：准确率、结论反转率、工具调用成功率、报告完整率、延迟、成本、人工采纳率、长尾 case 解决率。

### 标准答案方向

优先选 C：

> 我会优先选 C：保持当前模型，加强工具、规则、Eval 和数据治理。原因是这个任务的核心是基于当前数据做可验证归因，不是开放式生成。当前系统最大的风险来自数据口径、工具调用、贡献率计算、RAG 误导、结论校验和线上治理。只换更强模型无法解决这些确定性问题，甚至可能在成本更高的情况下继续犯同类错误。

为什么不优先 A：

> 更强通用模型可能提升复杂理解和报告表达，但它不能替代指标口径、数据源一致性、确定性计算和回归评测。对于标准化异动分析，如果工具和规则没打牢，模型越强也可能只是更流畅地说错。A 适合在工具链稳定后，用于高复杂度、多轮规划、证据冲突判断。

为什么不优先 B：

> SFT / 微调依赖大量高质量、结构化、已标注的历史 case。当前历史报告可能口径不统一、标签不标准、样本覆盖不完整，直接微调容易学到旧经验偏差。而且微调更适合学习表达风格、流程偏好和领域术语，不适合替代实时数据归因和工具计算。

证明指标：

> 我会用同一套 Eval 对三种方案做小规模实验，比较二级归因准确率、证据充分率、报告完整率、结论反转率、人工采纳率、工具调用成功率、平均耗时、单次成本和高风险 case 回退率。如果 C 能以最低成本显著降低错误结论率和结论反转率，就是优先做 C 的证据。

改变选择的条件：

> 如果评估发现主要错误来自模型理解能力不足，例如复杂多工具规划失败、长上下文中漏掉关键证据、冲突证据判断明显弱，那我会考虑 A。  
> 如果已经积累大量高质量结构化 case，并且错误主要集中在领域术语理解、报告风格、固定流程遵循，那我会考虑 B。  
> 但只要主要问题仍是数据、工具、规则和评估体系，C 就应该优先。

一句话总结：**先把确定性系统打牢，再考虑模型升级；不要用更强模型掩盖工具和数据治理问题。**

## 问题 20：如何用 3 分钟讲清楚这个项目？

如果面试官让你用 3 分钟介绍这个项目，不能照着 STAR 材料念，要讲得像一个真正做过的人。

4. Badcase 讲对了，但表达不够锋利。核心 badcase 应概括为：让模型自由选指标 / 取数导致口径漂移和错误归因，后来把确定性环节工具化。
5. 结果缺少完整指标组。不能只说 99%，还要说 2-3 天到分钟级、14h/周、实时下钻 1 分钟、人工一致性 80%，并解释 99% 是 T+1 历史分析场景。

### 标准答案方向

可直接用于面试的 3 分钟版本：

> 我这个项目做的是广告收入异动分析 Agent。它值得做的原因很直接：广告是核心营收业务，但过去收入异动分析高度依赖人工。产运需要从 Adata 等平台取数，再按业务线、场景、广告位、竞价 / 合约、曝光、CPM、行业、广告主逐层拆解，一次完整分析通常要 2-3 天，而且不同同学的分析口径不一致。对于收入波动这种问题，定位越慢，业务风险和潜在赔付越高，所以我们希望把这套标准化但高频的分析流程沉淀成 Agent。
>
> 我先做的不是直接写 Prompt，而是拆人工 SOP。我把异动分析拆成四层：第一层看整体收入变化，第二层拆竞价和合约，第三层判断是曝光驱动还是 CPM 驱动，第四层再下钻到行业、OG、广告主和广告位。曝光侧按请求、返回、原始曝光、有效曝光漏斗分析；CPM 侧按用户行为、预估指标、行业预算、广告主结构和扣费相关指标做贡献拆解。这样做的目的是避免模型直接跳结论，而是让它按固定分析路径逐层验证。
>
> 架构上，我们基于 WorkBuddy 承载用户入口和任务运行，把异动分析封装成 Skill；底层通过 MCP 暴露数据查询、指标计算、看板校验、知识检索等工具。这里我做了一个很重要的边界划分：取数、指标计算、贡献率拆解这类确定性环节不让模型自由发挥，而是通过参数化 SQL 和工具完成；模型主要负责理解用户意图、选择下钻路径、结合数据证据生成报告。RAG 只用于补充业务口径、流量范围和历史 case，而且历史 case 只能作为参考，不能覆盖当前数据事实。
>
> 这个项目里最关键的 badcase 是一开始我们把很多广告基础知识直接喂给模型，希望它自己判断该取什么指标、怎么下钻。结果模型会出现字段选错、口径漂移、把历史经验当当前事实、甚至强行建立因果关系的问题。后来我推动把确定性流程工具化：模型只能选择工具和参数，贡献率由工具计算，报告输出后再用独立 judge 检查数据完整性、计算闭环、结论证据和报告完整性。如果数据缺失就补取数，如果结论和数据冲突就降级为候选原因，不输出确定结论。
>
> 最终效果上，T+1 历史异动分析场景下，结论准确率从 75% 提升到 99%；单次分析从原来的 2-3 天缩短到分钟级；实时根因下钻控制在 1 分钟内；每周释放大约 14 小时人工分析成本。实时场景因为数据回补和慢指标限制，AI 与人工分析一致性约 80%，所以我们会明确标注数据新鲜度、缺失指标和置信度。
>
> 我的反思是，这类 Agent 的核心不是让模型自由推理，而是给模型搭一个可验证的 harness：清晰的指标树、确定性工具、业务知识库、校验规则和 Eval 回归。尤其是广告异动分析这种高风险业务场景，模型越应该被放在边界内做它擅长的规划和解释，事实、计算和口径必须交给工具和规则来保证。

这一版的优势：

- 业务价值清楚：提效 + 收入风险。
- 你的动作清楚：拆 SOP、定边界、工具化、RAG、judge、eval。
- 技术和产品结合：不是只讲 Prompt。
- 指标可信：解释了 99% 和 80% 的适用场景。
- 反思有高度：模型边界和 harness。

面试使用建议：

> 不要逐字背，但必须按这个顺序讲：业务痛点 -> 人工 SOP 拆解 -> Agent 架构 -> 关键 badcase -> 效果 -> 反思。  
> 讲项目时少讲“我如何思考总分总”，多讲“我具体拆了哪条业务链路、定义了哪些工具边界、解决了什么 badcase、数据结果是什么”。

## 第 11-20 题暴露出的新增短板

### 1. 因果意识需要补强

不要轻易说“导致”。广告系统里 CTR、CPM、pCTR、预算、排序、流量结构高度耦合，很多结论只能说“贡献解释”或“候选原因”。如果没有实验、策略日志、明确系统变更和反事实验证，不要包装成因果。

### 2. 数据口径是高风险短板

第 16 题回答不及格，说明你需要系统补指标治理：

- metric_id
- 口径版本
- 数据源权威等级
- 数据新鲜度
- 血缘关系
- 指标适用场景
- 对账和冲突处理规则

这是广告策略产品面试的底层能力。

### 3. RAG 不能替代事实

你已经知道历史 case 不能覆盖当前数据，但还需要形成更工程化的表达：

> 当前数据 > 工具计算 > 规则口径 > 历史 case。

这条优先级必须进入 Prompt、工具输出、校验器和 Eval，而不是只写在文档里。

### 4. 处置 Agent 要保守

从分析到处置是风险跃迁。能否自动执行不取决于动作名称，而取决于影响半径、可逆性、置信度、审批权限和回滚能力。

### 5. 项目介绍要少抽象、多业务链路

3 分钟项目介绍里，不要讲太多通用方法论。面试官要听到：

- 业务为什么痛
- 你拆了什么链路
- 你定义了什么边界
- 你解决了什么 badcase
- 结果怎么量化
- 反思有什么产品判断
