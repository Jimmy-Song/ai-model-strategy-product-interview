# 广告收入异动分析 Agent：十道面试追问复盘

整理时间：2026-06-22  
适用岗位：AI 大模型策略产品经理 / Agent 产品经理 / 广告策略产品经理

## 总体判断

你的项目材料具备真实业务场景、明确 AI Agent 落地点、可量化结果和较强业务价值，但在真实面试中会被重点追问四类问题：

1. 指标定义是否严谨：例如“准确率 99%”到底评估什么、样本量多大、真值如何得到。
2. Agent 边界是否清楚：哪些由 LLM 负责，哪些必须由规则、工具和确定性代码负责。
3. 线上治理是否完整：灰度、监控、回滚、人工反馈、Eval 回归是否能闭环。
4. 架构理解是否扎实：WorkBuddy、MCP、Skill、RAG、Evals、模型替换等概念不能只会泛泛描述。

下面按今天已经追问的 10 道题逐题复盘。

## 问题 1：准确率 75% -> 99% 到底怎么定义？

### 面试官追问

你材料里写：

> 异动分析准确率由 75% 提升至 99%，AI 归因与人工分析一致性约 80%。

这两个数字之间有明显张力。如果归因一致性只有 80%，为什么准确率能到 99%？

需要回答：

- 评估对象是什么：异动识别、一级归因、二级归因，还是报告整体正确？
- 标注真值来自哪里？
- 样本量是多少，覆盖哪些业务线和时间范围？
- 99% 是离线评测、线上结果，还是某个子模块结果？
- 99% 和 80% 的关系是什么？

### 你的回答主要问题

你的回答能把“准确率”落到曝光 / CPM 归因，且解释 99% 是历史 T+1 分析、80% 是实时异动分析，这一点有效。

但存在三个硬伤：

1. 样本量表述不严谨。你说 3 月到 6 月、每天 3-10 条，总样本“千条左右”，按时间推算大概是 360-1200 条，跨度过大，显得没有掌握评测集规模。
2. “准确率”定义仍不够精确。没有说明是 Top1 命中、Top3 命中、一级归因命中、二级归因命中，还是多因子贡献排序正确。
3. 真值来源有偏差风险。历史产运报告本身可能有口径差异或错误，需要说明如何清洗、统一标签和人工仲裁。

### 标准答案方向

更好的回答应把准确率拆成多层：

1. L1 归因准确率：收入异动主要由曝光还是 eCPM / CPM 驱动。
2. L2 归因准确率：如果是 CPM，进一步判断是用户行为、行业预算、广告主结构、模型预估、扣费机制、策略规则等哪类因素。
3. L3 证据充分率 / 报告可采纳率：结论是否被关键指标支撑，是否能被产运直接采用。

可参考表达：

> 这里的 99% 不是指所有复杂根因都完全命中，而是指历史 T+1 异动分析场景下，Agent 输出的一级或二级核心归因与人工复盘结论一致的比例。我们把历史产运报告结构化为标签体系，例如曝光链路、CPM-用户行为、CPM-行业预算、CPM-广告主结构、CPM-预估模型、CPM-策略规则等。评测时要求模型输出主因、证据指标和贡献解释，只有主因命中且关键指标能支撑结论才算通过。实时场景因为当天数据不完整，归因链路缺少最终结算和部分广告主预算信息，所以一致性约 80%。

还应补充：

> 由于人工历史报告只是近似真值，我们做了标签标准化和人工抽检。对于多因子共同作用的 case，不强行要求单一 Top1，而是看主因排序和贡献解释是否合理。

## 问题 2：CPM 下降时，Agent 怎么判断主因？

### 面试官追问

如果某天收入下降 10%，曝光基本不变，CPM 下降 10%。同时：

- CTR 下降 3%
- CVR 下降 8%
- 某头部行业预算下降 15%
- pCTR 下降 5%
- 实际扣费 CPC 下降 6%

Agent 怎么判断主因？

需要回答：

- CPM 怎么拆成可计算因子？
- 每个因子对 CPM 下降的贡献怎么量化？
- 多因素同时变化时，怎么避免强行因果归因？
- 报告里怎么表达主因和不确定性？

### 你的回答主要问题

你能想到 eCPM 与 pCTR、pCVR、bid 相关，也能提到“自身变化”和“结构变化”，这是亮点。

但有四个问题：

1. 公式过于简化。广告 eCPM 不一定都能写成 `pCTR * pCVR * bid`，不同计费模式、竞价 / 合约、oCPX / CPC / CPM / CPA 都有不同口径。
2. 混淆实际 CTR / CVR 和 pCTR / pCVR。实际 CTR 是结果指标，pCTR 是模型预估指标，二者不一定同步变化。
3. 把排除 CTR / CVR 后的剩余部分统一归为行业预算，过度简化。剩余因素还可能包括竞争强度、出价策略、扣费机制、召回填充、排序策略、保护规则等。
4. “80% 主因阈值”没有解释来源，是业务经验、历史调参还是管理口径，需要说清楚。

### 标准答案方向

先按广告产品和计费模式拆开，再讲统一框架：

> 我不会直接让模型自由判断主因，而是先把收入拆成可解释链路。最上层是收入 = 曝光量 * eCPM / 1000。曝光基本稳定时，问题落到 eCPM。对竞价广告，eCPM 可以进一步拆成流量结构、候选广告结构、预估质量、出价 / 预算、排序与扣费机制等模块；在 oCPX 场景下，排序侧近似受 pCTR、pCVR、bid 或目标成本影响，但不同计费模式要使用不同公式，不能一刀切。

贡献量化可以这样回答：

> 量化上我会做两类拆解。第一类是指标贡献拆解，例如用对数差分、顺序分解或 Shapley 方法，把 eCPM 下降分摊到 pCTR、pCVR、bid、扣费率、行业结构等因子上。第二类是结构拆解，例如行业 / OG / 广告主维度，用 mix effect 和 within effect 区分“单个行业 eCPM 下降”与“低 eCPM 行业占比上升”。最终不是只看相关性，而是要求贡献率、方向一致性、链路证据三者同时成立。

避免强行归因：

> 如果多个因子贡献接近，或者关键链路数据缺失，Agent 不输出唯一主因，而输出“多因子共同驱动”或“需人工确认”。报告里给出主因置信度、贡献排序、缺失数据和下一步排查建议。比如主因贡献超过 60% 或 Top2 累计超过 80% 才给明确结论，否则降级为候选原因列表。

## 问题 3：哪些判断由模型负责，哪些由规则 / 工具负责？

### 面试官追问

你提到通过 Prompt 约束、反思机制、校验规则解决幻觉、强行因果、口径漂移。

需要回答：

- 数据查询和指标计算，是否允许 LLM 自己写 SQL 或自由选维度？
- 贡献率计算，是模型算，还是工具算？
- 根因判断中，模型到底起什么作用？
- 校验规则拦截什么？举真实例子。
- 如果模型结论和规则计算冲突，怎么处理？

### 你的回答主要问题

你说不允许 LLM 自己写 SQL，而是人工写好 SQL 让模型执行，这是对的方向。你也提到数据校验和结论完整性校验。

最大问题是：

> 你说贡献率由模型计算。

这在真实面试里很危险。贡献率属于确定性数值计算，不应该交给 LLM 自己算。LLM 可以负责规划、解释和报告生成，但关键数值计算必须由工具完成。

另外：

1. “人工写 SQL，LLM 执行”应更准确地表述为“预置参数化 SQL / 指标工具，LLM 选择工具和填参数”。
2. 用 CDP 抓线上看板做数据校验可以作为过渡方案，但长期更稳定的是统一指标服务、数据 API 或元数据血缘校验。

### 标准答案方向

核心原则：

> LLM 不负责事实和数值的最终裁决。LLM 负责规划、工具选择、解释和报告生成；确定性数据查询、指标计算、贡献率拆解、阈值判断和一致性校验由工具 / 规则完成。

可参考回答：

> 我把链路分成工具层、策略规则层和模型推理层。数据查询不允许 LLM 自由写 SQL，而是封装成参数化工具，例如按业务线、广告位、日期、竞价 / 合约拉取收入、曝光、eCPM、CTR、CVR、行业、广告主等指标。LLM 只能选择工具和填参数，不能自由拼字段。指标计算和贡献率计算也不交给模型，而是由确定性工具完成。模型主要负责三件事：决定下一步下钻路径；把工具输出的贡献排序和业务知识结合形成候选根因；生成可读报告，并说明证据、不确定性和下一步排查建议。

校验规则：

> 校验规则主要拦截三类问题：数据不一致、计算不闭环、结论无证据。例如，Agent 拉取的收入总额必须和线上看板在同口径下误差小于阈值，否则中止分析；报告中如果说 CPM 是主因，则必须给出 CPM 变化幅度、对收入变化的贡献率，以及下钻到 CTR / CVR / 行业预算 / 广告主结构的证据。

冲突处理：

> 如果模型结论和规则冲突，默认规则优先，Agent 不输出确定结论，而是降级为“候选原因 + 冲突点 + 需人工确认”。只有人工复盘确认规则覆盖不到新 case 后，才把这个 case 纳入知识库和规则迭代。

## 问题 4：RAG 怎么避免历史经验把模型带偏？

### 面试官追问

历史知识库里有一条案例：

> 某业务 CPM 下降，历史上通常是教育行业预算下降导致。

但今天教育行业预算没下降，反而是游戏行业占比上升且 eCPM 较低。RAG 可能把模型带偏。

需要回答：

- 知识库存什么？
- 检索召回用什么 key？
- 怎么避免历史 case 覆盖当前数据事实？
- 检索出来的知识是参考信息还是强规则？
- 如何评估 RAG 是正向提升，而不是引入旧经验偏差？

### 你的回答主要问题

你能把知识库分成“基础概念”和“历史 case”，并说明基础概念是强规则、历史 case 是参考信息，这是对的。

但问题在于：

1. 召回机制只说语义相似，不够可靠。异动分析需要结构化过滤，例如业务线、广告位、异常指标、竞价 / 合约、行业、归因类型。
2. 避免带偏只靠 Prompt 约束“必须由数据推导”，这太软，需要证据优先级和适用条件校验。
3. RAG 效果没有量化，只说“结论更定制化”，不是面试级评估指标。

### 标准答案方向

知识库分三类：

> 第一是强规则类，包括指标口径、业务线流量范围、维度映射、SQL 参数约束；第二是归因规则类，包括曝光链路、CPM 拆解、竞价 / 合约差异、异常阈值；第三是历史 case 类，包括历史异动原因、对应证据、适用条件和排除条件。历史 case 只作为参考，不作为结论依据。

召回机制：

> 检索不是纯语义召回，而是结构化过滤 + 向量召回。先用业务线、广告位、指标类型、竞价 / 合约、异常方向做 metadata filter，再在候选集合里做语义相似召回。历史 case 必须带适用条件，例如“教育行业预算下降且教育行业收入占比超过某阈值时适用”，否则只能作为背景信息。

避免带偏：

> Prompt 里明确证据优先级：当前数据 > 规则计算 > 基础口径 > 历史 case。历史 case 只能生成候选假设，不能直接生成结论。报告输出时，凡是引用历史 case 的结论，都必须验证当前数据是否满足适用条件；不满足则标记为“不适用”。

评估 RAG：

> 做 A/B 评估：无 RAG、只有规则 RAG、规则 + 历史 case RAG 三组。指标看一级归因准确率、二级归因准确率、证据充分率、幻觉率、平均分析耗时、人工采纳率，并单独统计“历史经验误导率”，即模型引用了历史 case 但当前数据不支持的比例。

## 问题 5：实时预警和 1 分钟下钻怎么设计？

### 面试官追问

如果今天 14:05 收入突然下降，Agent 要在 1 分钟内给出初步根因，链路怎么跑？

需要回答：

- 预警触发条件是什么？
- 实时数据延迟和不完整性怎么处理？
- 1 分钟内能下钻到什么深度，哪些要异步补充？
- 为了速度，预计算哪些指标？
- 14:05 报警，14:10 数据回补后结论反转，产品上怎么设计？

### 你的回答主要问题

你答到了告警等级触发、数据延迟标注、先出初步结论、回补后重跑，这是对的。

不足在于：

1. 触发条件太粗，只说同比 / 环比明显变化，没有说明基线建模、节假日、小时周期和误报控制。
2. 1 分钟下钻到广告主 / 广告位需要预聚合，否则实时拉明细不稳定。
3. 结论反转的产品设计不完整，没有提版本化、置信度、通知更新、审计日志。

### 标准答案方向

预警触发：

> 实时告警不是简单同比 / 环比，而是基于同星期同小时、近 N 天移动均值、节假日修正后的预测基线。触发条件可以是收入、曝光、eCPM 偏离预测值超过阈值，并且连续多个时间窗口成立，再按影响金额和偏离幅度分级。只有 P0 / P1 或连续告警才自动触发 Agent 下钻。

实时数据处理：

> 实时场景要区分快指标和慢指标。收入、曝光、请求、填充、pCTR、pCVR、出价、行业消耗可以分钟级；转化、ROI、深度转化等慢指标只作为待确认因素。报告必须带数据新鲜度和完整度，例如“截至 14:05，收入数据延迟 2 分钟，转化数据未完全回传”。

1 分钟下钻：

> 1 分钟内只做初步根因，不追求最终结论。下钻深度到业务线、广告位、竞价 / 合约、Top 行业、Top 广告主、曝光 vs eCPM。需要预计算分钟级收入、曝光、eCPM、行业 TopK、广告主 TopK、CTR / CVR / pCTR / pCVR、预算消耗、库存和请求漏斗。慢指标和长周期转化归因异步补充。

结论反转：

> 产品上要做结论版本管理。14:05 输出 v1 初步诊断，标记置信度和缺失指标；14:10 数据回补后自动生成 v2，并明确“结论是否变化、变化原因、旧结论是否失效”。如果旧结论影响了人工操作，需要通知相关群或工作台，并保留审计日志。

## 问题 6：项目除了提效，有没有商业价值？

### 面试官追问

你说释放 14h/周人工成本，并把 2-3 天缩短到分钟级。但这只是提效，不一定创造业务价值。

需要回答：

- 除了节省人力，还有没有直接或间接收入收益？
- 收益如何量化？
- 如果没有直接收入收益，怎么证明值得做？
- 北极星指标和过程指标是什么？
- 如果只能向老板汇报一个指标，选哪个？

### 你的回答主要问题

你能把价值上升到“减少异常期间收入损失 / 广告主赔付损失”，并考虑 Agent 成功率折损，这是加分点。

但两个问题比较明显：

1. “员工成长空间”不是强商业论证。可以作为组织效率补充，但老板更关心收入风险、SLA、规模化复用、口径一致性、专家经验沉淀。
2. 北极星指标选“准确率”不一定最优。准确率是门槛，但只看准确率会诱导系统少报、慢报、只报简单 case。

### 标准答案方向

商业价值：

> 这个项目的价值不只是节省 14h/周，而是降低广告收入异常的 MTTA 和 MTTR。以前从发现异动到定位原因可能要 2-3 天，现在分钟级给出初步根因，核心收益是减少异常持续时间、降低广告主赔付风险、提升收入波动治理能力。

收益量化：

> 业务价值 = 异常影响金额 / 分钟 * Agent 缩短的定位时间 * 归因可采纳率。  
> 如果涉及广告主赔付，则进一步计算：历史单位时长赔付金额 * 缩短时长 * 归因成功率。  
> 同时可以量化异常发现率、平均定位时长、人工介入率、报告采纳率，以及因提前发现而避免的潜在损失。

如果短期没有直接收入：

> 它仍然是风控和经营分析基础设施。价值包括：异常排查标准化、降低口径不一致、沉淀专家经验、减少对少数人的依赖、多业务线复用、为后续自动预警 / 自动诊断 / 自动处置打基础。

指标体系：

> 北极星指标可以选“可采纳根因覆盖率”或“有效异动处理时长下降”。过程指标包括告警召回率、误报率、平均检测延迟、平均下钻耗时、一级 / 二级归因准确率、证据充分率、人工采纳率、结论反转率、人工介入率、单位 case 成本。

向老板只汇报一个指标：

> 我会选“P0/P1 异动平均定位时长下降”，因为它最容易和业务风险、收入损失、广告主赔付建立关系。准确率必须作为门槛指标，例如二级归因准确率超过 95% 后，再汇报定位时长下降。

## 问题 7：灰度上线和线上监控怎么保证不大规模推错结论？

### 面试官追问

Agent 要上线给多个业务线使用，如何设计灰度上线和线上监控？

需要回答：

- 上线前离线评测集怎么构建？
- 灰度阶段怎么分流？
- 上线后监控哪些指标？
- 出错时怎么回滚或降级？
- 如何设计人工反馈闭环？

### 你的回答主要问题

你能按曝光漏斗、CPM 驱动因素、不同业务线和流量场景构建评测集，这是对的。

但问题包括：

1. 评测集缺少难度和风险分层，例如多因子、数据缺失、指标冲突、历史 case 误导、低样本波动、节假日周期。
2. 灰度不能只按业务线，还需要只读模式、人工确认模式、自动推送模式。
3. 上线门槛没有具体标准。
4. 你对“错误结论怎么回滚或降级”一开始答不出来，这是线上 AI 产品的大风险点。

### 标准答案方向

离线评测集：

> 上线前构建分层评测集。第一层是常规单因子 case，包括曝光漏斗、CPM、CTR / CVR、pCTR / pCVR、行业预算、广告主结构。第二层是复杂 case，包括多因子共同变化、贡献率接近、数据延迟、口径缺失、历史 case 容易误导、合约和竞价混合。第三层是高风险 case，来自历史 P0/P1 异常和人工复盘报告。每个 case 都有标准标签、关键证据和可接受结论范围。

灰度分流：

> 灰度分三阶段：只读影子模式、人工确认模式、自动推送模式。第一阶段 Agent 跟人工并跑，不影响业务决策；第二阶段结论给产运确认后再同步；第三阶段只对高置信、规则闭环的 case 自动推送。分流上按业务线、异常类型和风险等级逐步扩大。

线上监控：

> 监控分五类：链路稳定性、数据质量、结论质量、风险指标、业务效果。  
> 链路稳定性包括任务成功率、工具调用成功率、API 成功率、平均耗时、超时率、重试率。  
> 数据质量包括数据新鲜度、数据完整率、指标服务一致率、核心字段缺失率。  
> 结论质量包括一级 / 二级归因准确率、证据充分率、报告完整率、人工采纳率、结论反转率。  
> 风险指标包括错误结论率、高风险 case 自动推送率、人工拦截率、低置信结论占比。  
> 业务效果包括 P0/P1 平均定位时长、异常发现到确认时长、人工介入时长、重复问题解决率。

回滚 / 降级：

> Agent 要设计多级降级，而不是只有开关。第一级是结论降级：数据缺失、工具失败、贡献率不闭环时，不输出确定主因，只输出候选原因和需人工确认项。第二级是能力降级：如果 RAG 或模型不稳定，只保留确定性规则报告，例如收入、曝光、CPM、TopK 变化，不做自然语言归因。第三级是流量降级：某业务线错误率超过阈值，自动退回影子模式或人工确认模式。第四级是版本回滚：Prompt、工具、规则、知识库都要版本化；发布后错误结论率或结论反转率异常升高，回滚到上一版。

人工反馈闭环：

> 反馈不能只是“好 / 不好”，要结构化到错误类型：数据错误、维度选错、计算错误、主因判断错、证据不足、报告不完整、知识引用错误。反馈后不应直接写入 RAG，而是进入人工审核和标签化流程。通过审核后，分别更新规则、工具、Prompt、知识库或评测集。不是所有错误都适合靠 RAG 解决，计算错要修工具，字段口径错要修指标服务，归因边界错才更新规则或知识库。

## 问题 8：如何设计 Agent Eval 体系？

### 面试官追问

如果要把广告异动分析 Agent 做成公司内通用能力，不只服务 CSIG，怎么设计 Agent Eval？

需要回答：

- Eval case 怎么采样？人工历史报告够不够？
- 评分维度有哪些？
- 报告生成质量怎么评分？
- LLM-as-judge 能不能用？怎么避免 judge 胡说？
- Eval 怎么接入日常迭代，防止 Prompt / RAG / 工具升级后效果回退？

### 你的回答主要问题

你把稳定性、数据质量、结论质量、业务效果这些维度列出来了，但有几个关键问题：

1. 把线上过程指标当成 Eval case。工具调用率、接口成功率是监控指标，不是 eval case 本身。
2. 漏答“报告生成质量怎么评分”。
3. LLM-as-judge 的定位不准确。Judge 适合评估报告结构、证据引用、遗漏、矛盾，不适合评估接口成功率这类确定性指标。
4. “隔离上下文”不能防止 judge 胡说。Judge 需要受控输入，而不是信息越少越客观。

### 标准答案方向

Eval case 定义：

> Eval case 不等于线上日志指标。一个 eval case 应该包括：用户问题、业务线和时间范围、冻结的数据快照、工具调用期望、标准归因标签、关键证据、允许的等价答案、评分 rubric。人工历史报告可以作为初始来源，但不够，因为它偏向已发生且被记录的 case，还要补充人工构造的边界 case、数据缺失 case、多因子 case、历史经验误导 case，以及高风险 P0/P1 case。

评分维度：

> 分四层评分：任务完成、数据正确、归因正确、报告可用。任务完成看是否成功跑完、是否调用正确工具、耗时是否达标；数据正确看口径、时间范围、业务线、维度选择、计算结果是否正确；归因正确看一级归因、二级归因、贡献排序、证据闭环、不确定性表达；报告可用看结构完整、结论先行、数据支撑、风险提示、下一步建议、是否可被产运直接采用。

报告质量：

> 报告生成质量不是看文笔，而是看业务可用性：是否回答了收入变动幅度、主因、贡献率、影响范围、证据指标、置信度、待确认项和建议动作。可以人工评分 1-5 分，也可以让 LLM judge 做初筛，但关键 case 仍然人工抽检。

LLM-as-judge：

> LLM-as-judge 可以用，但只评估半结构化内容，比如报告是否引用证据、结论是否和给定数据矛盾、是否遗漏必填项。为了避免 judge 胡说，要给它固定 rubric、标准答案、数据摘要和 Agent 报告，并要求输出结构化评分和引用依据。关键准确率指标仍以确定性规则和人工标注为准，同时定期评估 judge 与人工的一致性。

接入日常迭代：

> 每次 Prompt、RAG、工具、指标口径升级，都跑固定回归集 + 最新线上失败集。设置 blocking gate，例如二级归因准确率不能下降超过 1pct，高风险 case 不能回退，数据一致率必须 100%。线上 badcase 经过标注后进入 eval 集，形成持续回归。

## 问题 9：WorkBuddy、MCP、Skill 的架构边界是什么？

### 面试官追问

你项目中用了 WorkBuddy + MCP + Skill。

需要回答：

- WorkBuddy、MCP、Skill 分别承担什么角色？
- 为什么需要 MCP？不用 MCP 直接调 API 不行吗？
- Skill 怎么定义和复用？
- 多业务线复用时，哪些通用，哪些定制？
- 如果换模型，哪些不变，哪些要重测？

### 你的回答主要问题

这题暴露比较明显：

1. WorkBuddy / MCP / Skill 边界讲混了。
2. 说 MCP 是“封装的一整套与大模型通信的 API 服务”，不准确。MCP 是把外部工具、数据源、资源以标准协议暴露给模型 / Agent 使用的连接层。
3. 说 MCP 比直接调 API 更快，这个理由不稳。MCP 不天然更快，价值在标准化、可复用、权限治理、工具发现、上下文协议统一。
4. Skill 复用没答出来。
5. 换模型后哪些要重测没答出来。

### 标准答案

架构角色：

> WorkBuddy 是用户入口和 Agent 运行平台，负责承载用户交互、任务触发、权限、执行记录和结果展示。MCP 是工具连接层，把数据库查询、指标计算、看板校验、知识检索等能力以标准工具形式暴露给 Agent。Skill 是任务级能力封装，定义“什么时候调用、按什么步骤执行、能用哪些工具、输出什么格式、如何校验结果”。

为什么需要 MCP：

> 不用 MCP，Agent 当然也可以直接调后端 API，但会有工具接入方式不统一、权限和审计难管、工具描述和参数 schema 分散、不同 Agent 或不同模型复用成本高等问题。MCP 的价值是把工具能力标准化，让模型通过统一协议发现和调用工具，同时便于权限、日志、版本和复用治理。它不是为了让调用更快，而是为了让工具生态更可控、更可扩展。

Skill 定义：

> 一个 Skill 至少包括：适用场景 description、触发条件、输入参数、分析流程、可调用工具列表、业务规则、输出模板、异常处理、校验规则和示例 case。比如 `csig_revenue_analysis` 会定义：先确认业务线和时间范围，再拉收入 / 曝光 / eCPM，再做竞价 / 合约拆分，再做曝光或 CPM 下钻，最后输出主因、贡献率、证据和不确定性。

多业务线复用：

> 通用部分是收入异动分析框架、指标计算工具、贡献率拆解、报告模板、校验规则和 eval 体系。定制部分是业务线流量范围、广告位映射、特殊策略规则、行业分类、数据源字段、历史 case 和阈值。产品化上应该把通用流程做成 base skill，把业务差异做成配置和知识库，而不是复制多份 skill。

换模型时：

> 架构上工具层、指标计算、数据 API、规则校验、权限审计、评测集不需要变。需要重测的是模型的工具调用稳定性、指令遵循、长上下文能力、RAG 使用方式、报告生成质量、对不确定性的表达、以及复杂多步任务完成率。推理模型可能更适合复杂归因，但成本和延迟更高，所以要重新评估准确率、耗时和成本 trade-off。

## 面试临场建议

1. 先定义口径，再讲结果。不要一上来抛 99%、80%、分钟级这类数字，必须先说明指标定义。
2. 少说“模型自己判断”，多说“模型调用工具、规则校验、人工反馈闭环”。
3. 遇到不确定的问题，不要硬编。可以说“这里我会降级为候选原因，不输出唯一主因”。
4. 讲 AI 项目时，不要过度迷信后训练。这个项目的核心价值是工具化、可验证、可治理，而不是让模型自由推理。
5. 每个项目结果都尽量翻译成业务语言：MTTA、MTTR、P0/P1 定位时长、赔付风险、报告采纳率、人工介入率。
